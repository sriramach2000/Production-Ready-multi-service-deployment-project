== 1. Executive Summary

This project is a **production-grade, containerized Task Management REST API** built with FastAPI, PostgreSQL, Redis, and Celery — orchestrated via Docker Compose with Nginx reverse proxying and Prometheus/Grafana monitoring.

The entire project is driven by a single source of truth: `orchestrate.py` (~1,250 lines). This one file holds all configuration, generates 29 project files from templates and inline generators, and provides a CLI to manage the Docker stack.

**Key design principles:**

* **Single source of truth** — change a config value once, it propagates to every file that uses it
* **Two kinds of files** — application code lives in `templates/` (editable with full IDE support); infrastructure config is generated from inline functions
* **Production patterns** — multi-stage Docker builds, non-root containers, health checks, named volumes, restart policies, environment variable management

=== Step-by-Step Walkthrough

[arabic]
. **Developer edits** `templates/api/app.py` (application code) or `CONFIG` in `orchestrate.py` (configuration)
. **Developer runs** `python orchestrate.py up` (or `generate` for files only)
. **orchestrate.py** loads the `CONFIG` object (Section 1)
. **Template rendering** — Jinja2 processes `templates/*.py` files, replacing `<< marker >>` syntax with CONFIG values
. **Generators execute** — inline functions produce Dockerfiles, docker-compose.yml, nginx configs, monitoring configs, etc.
. **File manifest** — `get_file_manifest()` assembles all 29 (path, content, description) tuples
. **Disk I/O** — `write_generated_files()` creates directories and writes all files
. **Docker Compose** — `docker compose up -d --build` builds images and starts the 7-service stack
. **Health checks** — Docker verifies each service is healthy before dependents start
. **Stack is live** — API accessible at `http://localhost:80` via Nginx

**Why custom delimiters?** Standard Jinja2 uses `{{ }}` which conflicts with Python dicts, f-strings, and other syntax. The `<< >>` delimiters allow template files to be valid Python that IDEs can parse, lint, and autocomplete.

**Key functions:**

* `render_template(template_path, config, header_lines)` — core rendering function
* `_template_header(template_path)` — generates the "GENERATED by orchestrate.py" header
* Convenience wrappers: `generate_api_app()`, `generate_worker_app()`, `generate_conftest()`, `generate_test_tasks()`, `generate_tests_init()`, `generate_alembic_env()`

=== 5.1 Marker Syntax & Convention

Templates use `<< variable >>` syntax — custom Jinja2 delimiters configured in `orchestrate.py`:

[source,python]
----
template_env = Environment(
    variable_start_string="<<",   # << postgres.user >>
    variable_end_string=">>",
    block_start_string="<%",      # <% if debug %> ... <% endif %>
    block_end_string="%>",
    comment_start_string="<#",    # <# This is ignored #>
    comment_end_string="#>",
)
----

**Context passed to templates:**

[source,python]
----
context = {
    "postgres": config.postgres,   # PostgresConfig instance
    "redis": config.redis,         # RedisConfig instance
    "api": config.api,             # ApiConfig instance
    "nginx": config.nginx,         # NginxConfig instance

**Key architectural decisions in the API:**

* **Async everywhere** — `asyncpg` driver, `AsyncSession`, `async def` routes
* **Dependency injection** — `get_session()` as FastAPI `Depends()` parameter
* **Cache-aside pattern** — Check Redis first on GET, invalidate on UPDATE/DELETE
* **Pydantic v2** — `from_attributes=True` for ORM model serialization
* **Single file** — All code in one `app.py` for simplicity in a containerized context

**Task ORM Model:**

[cols="2,2,2,2", options="header"]
|===
| Column | Type | Constraints | Default

| `id` | Integer | Primary key, autoincrement | —
| `title` | String(255) | NOT NULL | —
| `description` | Text | nullable | NULL
| `status` | Enum(TaskStatus) | — | `todos`
| `priority` | Enum(TaskPriority) | — | `MEDIUM`
| `created_at` | DateTime | — | `func.now()`
| `updated_at` | DateTime | nullable | `onupdate=func.now()`
|===

**API Endpoints (mounted at `/api/v1/tasks`):**

[cols="1,2,3,1", options="header"]
|===
| Method | Path | Description | Status

| POST | `/` | Create a new task | 201
| GET | `/` | List tasks with filters + pagination | 200
| GET | `/{task_id}` | Get single task (cache-aside) | 200/404

Additionally, `GET /` at the root serves as a health check returning `{"status": "ok"}`, and `GET /metrics` exposes Prometheus metrics.

A ~120-line single-file Celery application with 2 sections.

**Section 1: Celery Configuration**

[source,python]
----
celery_app = Celery("worker")
celery_app.conf.broker_url = os.getenv("CELERY_BROKER_URL", "<< redis.celery_broker_url >>")
celery_app.conf.result_backend = os.getenv("CELERY_RESULT_BACKEND", "<< redis.celery_result_backend >>")
celery_app.conf.task_serializer = "json"
celery_app.conf.result_serializer = "json"
celery_app.conf.accept_content = ["json"]
celery_app.conf.timezone = "UTC"
----

**Section 2: Task Definitions**

[cols="2,3,2", options="header"]
|===
| Task | Purpose | Binding

| `generate_report(filters)` | Long-running report of tasks matching filters | `bind=True` (for retries)
| `bulk_status_update(task_ids, new_status)` | Batch update multiple task statuses | `bind=True`
|===

IMPORTANT: The worker uses **synchronous** SQLAlchemy with `psycopg2-binary` — not `asyncpg`. Celery is process-based and does not run an async event loop. This is why `worker/requirements.txt` differs from `api/requirements.txt`.

**Startup timeline:**

[arabic]
. PostgreSQL and Redis start first (no dependencies)
. Health checks run: `pg_isready -U taskapp` every 10s (5 retries) and `redis-cli ping` every 10s (5 retries)
. Once both are healthy → API and Worker containers start
. API health check: HTTP GET to `localhost:8000/` every 30s (3 retries)
. Once API is healthy → Nginx starts and begins proxying traffic
. Prometheus and Grafana start independently in parallel with everything else

=== 6.9 Multi-Stage Dockerfile Pattern

Both API and Worker Dockerfiles follow the same multi-stage pattern:

....
┌─────────────────────────────────────────────────┐
│  STAGE 1: Builder                               │
│  ─────────────────                              │
│  FROM python:3.12-slim AS builder               │
│  WORKDIR /app                                   │
│  COPY requirements.txt .                        │
│  RUN pip install --prefix=/install -r ...       │
│                                                 │
│  (Dependencies installed to /install)           │
│  (Leverages Docker layer caching)               │
├─────────────────────────────────────────────────┤
│  STAGE 2: Production                            │
│  ────────────────────                           │
│  FROM python:3.12-slim                          │
│  RUN addgroup --system appgroup &&              │
│      adduser --system --group appuser           │
│  WORKDIR /app                                   │
│  COPY --from=builder /install /usr/local        │
│  COPY ./app.py ./app.py                         │
│  USER appuser            ← non-root!            │
│  EXPOSE 8000                                    │
│  HEALTHCHECK ...         ← container health     │
│  CMD ["uvicorn", ...]    ← or celery worker     │
└─────────────────────────────────────────────────┘
....

**Benefits:**

* **Smaller images** — build tools and pip cache not in final image
* **Layer caching** — `requirements.txt` changes rarely, so the `pip install` layer is cached
* **Non-root user** — `appuser` runs the process (security best practice)
* **Health checks** — Docker monitors container health and restarts if unhealthy

. Service layer queries PostgreSQL (and optionally Redis cache)
. Response flows back through middleware (duration recorded) → Nginx → Client

== 10. Security Architecture

=== Non-Root Containers

Both API and Worker Dockerfiles create a dedicated system user:

[source,dockerfile]
----
RUN addgroup --system appgroup && adduser --system --group appuser
USER appuser
----

The application process runs as `appuser`, not `root`. This limits the blast radius if the container is compromised.

=== Network Isolation

* All services communicate on the `app-network` bridge network
* Only Nginx exposes port 80 to the host (production)
* Internal services (API, PostgreSQL, Redis, Worker) are not directly accessible from outside the Docker network in production
* Development overrides expose additional ports for debugging convenience

=== Environment Variable Management

* Secrets are stored in `.env` (generated, never committed)
* `.env` is listed in `.gitignore`
* `.env.example` provides a safe-to-commit reference template
* Docker Compose loads env vars via `env_file: .env`
* No hardcoded credentials in application code or Dockerfiles

=== Nginx Security Headers

The following security headers are active on all proxied responses (configured in `generate_nginx_default_conf()`):

* `X-Frame-Options: DENY` — prevents clickjacking
* `X-Content-Type-Options: nosniff` — prevents MIME sniffing
* `X-XSS-Protection: "1; mode=block"` — XSS filter for older browsers
* `Referrer-Policy: strict-origin-when-cross-origin` — limits referrer information

All headers use the `always` directive to ensure they are sent regardless of response status code.

=== CORS Middleware

FastAPI's `CORSMiddleware` is configured in Section 9 of the API template to control cross-origin request access.

== Troubleshooting Guide

=== Common Issues

[cols="3,4", options="header"]
|===
| Problem | Solution

| `docker compose` command not found | Install Docker Desktop or Docker Engine v20.10+
| Port 80 already in use | Change `nginx.external_port` in CONFIG, regenerate
| PostgreSQL health check failing | Check `docker compose logs postgres` for errors
| API can't connect to PostgreSQL | Ensure `POSTGRES_HOST=postgres` (Docker service name, not localhost)
| Redis connection refused | Check Redis health: `docker compose exec redis redis-cli ping`
| Celery worker not picking up tasks | Verify `CELERY_BROKER_URL` matches between API and Worker
| Templates not reflecting changes | Run `python orchestrate.py generate` to re-render
| Hot reload not working | Ensure `docker-compose.override.yml` exists (dev mode)
| Grafana shows no data | Verify datasource UID matches (`uid: prometheus` in `datasource.yml` must match dashboard JSON). If UID was changed, delete the Grafana volume and recreate. Also check Prometheus is scraping: `curl http://localhost:9090/api/v1/targets`
|===

=== Useful Debug Commands

[source,bash]
----
# Check all service status
python orchestrate.py status

# View logs for a specific service

python orchestrate.py logs worker

# Shell into the API container
make shell

# Run a one-off command in the API container
docker compose exec api python -c "from app import settings; print(settings.DATABASE_URL)"

# Check database connectivity
docker compose exec postgres psql -U taskapp -d taskdb -c "SELECT 1;"

# Check Redis connectivity
docker compose exec redis redis-cli ping

# Inspect Celery worker
docker compose exec worker celery -A app inspect active

# Full reset (destroy all data)
python orchestrate.py clean
python orchestrate.py up
----
